## 결정 사항 (사용자 확인 완료)

| 항목 | 결정 |
| --- | --- |
| 검수 방식 | **규칙 기반 매칭만** (LLM 미사용) |
| 백엔드 | **가벼운 로컬 백엔드** (Node + Express) |

| 준수기준 입력 | **화면 내 용어 매핑 테이블**(원문어 → 번역어) 직접 입력 |
| 배포 | 로컬만. 로그인 없음 |
| 디자인 | 다크 미니멀. 원문/번역문 박스는 흰 바탕 검정 글씨 |

## 아키텍처

```
[브라우저 프론트엔드]  --docx 업로드/검수요청-->  [Express 백엔드]
   index.html              POST /api/check          mammoth: docx → 단락 배열
   style.css                                          규칙 매칭 엔진
   app.js                  <--검수결과(JSON)---       결과 반환
```

- 빌드 스텝 없음(번들러 X) → `npm install && npm start` 만으로 로컬 실행.
- 백엔드: Express + multer(파일 업로드) + mammoth(docx 파싱).
- 프론트: 정적 HTML/CSS/바닐라 JS. Express가 정적 서빙.

## 화면 (spec §4 — 사분 구성)

```
┌─────────────────────┬─────────────────────┐
│  원문 (.docx 업로드)  │ 번역문 (.docx 업로드) │  ← 상단 좌우
│                     │            [검수 시작]│
├─────────────────────┴─────────────────────┤
│ 번역 준수 기준 — 용어 매핑 테이블             │  ← 하단
│ [원문어] → [번역어]  [+행 추가]               │
└───────────────────────────────────────────┘
```

- 검수 후: 원문/번역문에서 **미준수 단락**만 노란 형광펜 배경으로 표시, 좌우 나란히 대응.
- 번역문 미준수 항목은 인라인 편집 가능(textarea/입력창).
- spec §9 DoD: 더 이상 수정사항 없으면 원문/번역문 미표시(통과 메시지).

## 규칙 매칭 엔진 (규칙 기반)

단락 인덱스 정렬 가정(번역물은 원문 단락과 1:1 평행 구조가 일반적).

1. **용어 매핑 규칙**: 각 (원문어 → 기대 번역어) 쌍에 대해
   - 원문 단락이 `원문어` 포함 && 대응 번역문 단락이 `기대 번역어` 미포함 → 해당 단락 쌍 미준수로 표시.
2. **용어 통일 규칙**: 같은 원문어가 번역문에서 2개 이상 서로 다른 번역으로 등장하면 불일치로 표시(ENS/NCTS 사례).
3. 매칭은 대소문자 무시, 부분 문자열 기준.

> 문체(평서체)·맥락(답변만 번역) 규칙은 규칙 기반 범위 밖 → out of scope (LLM 선택 시 후속).

## 예외 처리 (spec §3)

- 지원 안 하는 형식(PDF 등) 업로드 → 안내 메시지("docx만 지원").
- 대용량 파일 → 업로드 크기 제한 + 처리 중 로딩 표시. 파싱 실패 시 에러 안내.
- 원문/번역문 단락 수 불일치 → 경고 표시 후 가능한 범위로 매칭.

## 범위 제외 (spec §8)

회원가입, 다국어, 모바일, 문체/맥락 자동판정(LLM 영역).
