### 채점 기준

필수 구현 사항
1. 원문, 번역문 간의 단락, 문장의 매칭이 잘 되는가? (단락 수, 문장 수, 단락 길이, 문장 길이)
   - 원문의 단락 수(x1)와 번역문의 대응 단락 수(y1) 차이의 크기는?
   - 원문의 문장 수(x2)와 번역문의 대응 문장 수(y2) 차이의 크기는?
   - 원문의 각 단락 행수(x3)와 번역문의 대응 단락 행수(y3) 차이의 크기는?  
   - 원문의 각 문장 길이(행수,x4)와 번역문의 대응 문장 길이(행수,y4)는 몇 행 차이가 있나?
     . V=y/x*100, 
         x=x1,x2,x3,x4 
         y=y1,y2,y3,y4
     . bad : V<90 or >110
     . fair : 90=<V<95, 105<V=<110
     . good : 95=<V=<105          
  
2. 용어집 기준(번역 준수 기준)으로, 번역문의 용어가 용어집의 번역어로 정확히 적용되었나?
   - 원문의 용어(STterm)에 대해 번역문에서는 몇 개의 대응표현을 번역어(TTterm)
     로 사용했나? 다의어 번역이 몇 개인가?
   - 용어 번역 정확성 검토 프로세스
     . 용어집에서 좌측열 원문의 용어는 항목별로 한 개씩 입력한다.
     . 용어집에서 우측열 번역문의 용어는 좌측열 원문의 용어에 대해 한 개씩 입력한다.
      
     <process 1> TYPE 기준
      TYP|{STterm}|=A, A=1
      TYP|{TTterm}|=A', A'=0,1,2,3,...
     . if A=1 & A'=1, then good
       if A=1 & A'=2, then fair
       if A=1 & A'=0 or >=3, then bad
       * fair, bad는 반드시 번역가의 수정작업을 거치도록 한다.
     <process 2> TOKEN 기준
      TYP|{STterm}|=A, A=1
      TOK|{STterm}|=a, a=0,1,2,3,...
      TOK|{TTterm}|=a', a'=0,1,2,3,...
     . if a=a', then good
       if a>a' or a<a', then bad
       * bad의 경우 반드시 번역가의 수정작업을 거치도록 한다.
       
   - 원문의 용어에 대해 번역문에서 용어집을 준수하지 않은 표현/준수한 표현의 갯수와 비율은?
     . 용어집의 용어와 원문의 용어가 동일하더라도, 번역 준수 기준으로서 용어집의 용어의 의미와 
      별개의 의미인 경우(번역가가 검수한 결과)는 번역의 정확도 측정에서 제외한다.
       ex) <제외사례> *. 번역 준수 기준과 별개의 의미로, 원문에서 사용된 경우 
           cover - 용어집: (외부)덮개, 원문: "포함하다" 의미로 사용됨.
           <정확도 측정시 포함사례> *. 다의어 번역의 경우
           establishment - 용어집: 작업장, 번역문: 사업장으로 번역됨.
           이 부분은 번역물에 대해 웹 앱에서 표시한 사례들을 번역가가 검수한 후 해당 갯수를 결정
           하여, 측정정확도 산출시 보정하여 입력하도록 함. 
       if A=1 & a/a'=1, then good
       if A=1 & a/a'>1 or <1, then bad    
           
                               
3. 원문의 표현이 번역문에도 그대로 사용되었나? 해당 갯수를 번역가가 웹 앱을 통해 확인하여, 측정
   정확도 산출시 보정하여 입력하도록 함.
   - 번역 누락/미번역의 케이스인가? 해당 갯수는 몇 개? 정확도 판정 기준 수치는?
   - 번역문에서 원문을 그대로 사용하는 경우, 번역문 다음에 원문의 원어 표현을 병기하였나? 해당 
     갯수는 몇 개? 정확도 판정 기준 수치는?
     해당 용어가 번역문에서 병기된 경우. 원문 상으로 맨처음에 등장하는 해당 용어를 
     기준으로, 번역문에서 그에 대응하여 처음 등장하는 경우만 병기 허용. 이후 병기 시 불필요한 
     병기로 분류함.

## 현재 구현 결과 요약

- `server.js`는 `.docx` 및 `.xlsx` 용어집을 모두 지원하며, `verification` 폴더에서 파일명에 `용어집` 또는 `준수기준`이 포함된 것만 로드합니다.
- 용어집은 같은 원문 용어의 여러 번역어를 `/`로 병합하여 다의어를 허용합니다.
- `public/index.html`에는 원문/번역문 업로드 폼과 검수 버튼, 용어집 표, 결과 영역이 있으며, 검수 결과는 좌우 대응 테이블 형태로 렌더됩니다.
- `public/app.js`는 미준수 항목을 `mark`로 하이라이트하고, 번역문 셀을 `contentEditable`로 만들어 인라인 편집을 지원합니다.
- 검수 완료 시 `export` 버튼이 표시되어 수정본 `.docx` 내보내기를 시도할 수 있습니다.
- 잘못된 파일 형식, 누락된 파일, 텍스트 추출 실패 등의 상황에서 에러 메시지를 반환하며, 앱이 비정상 종료되지 않습니다.

## 추가 확인/개선 포인트

- 현재 UI는 검수 결과를 좌우 대응 표로 표시하지만, spec 상의 "원문/번역문을 화면에 좌우 나란히 배치"를 더 명확히 하려면 결과 영역 외에 별도 원문·번역문 뷰를 추가하는 것이 좋습니다.
- `verification` 폴더 용어집 데이터 품질에 따라 검사 정확도가 달라지므로, 용어집 파일에 명확한 `note` 지시문과 통일 규칙을 포함해야 합니다.
- `golden_set.md`는 이 구현 상태를 기준으로 검증합니다. UI/요구 사항 변경 시 다시 갱신해야 합니다.
